# 产品愿景

> **gm-agent** - 可持久化的自主 Agent 执行引擎

---

## 🎯 核心定位

**gm-agent 不是聊天机器人，而是自治执行引擎。**

当 AI 模型不再是一问一答，而是需要：
- 持续运行跨分钟/小时的任务
- 被外部事件主动触发（用户/文件/定时器/webhook）
- 支持多请求者并发插入任务
- 可中断、可恢复、可审计、可回滚

这时需要的是 **事件驱动的 Durable Runtime**，而不是 while-loop 自由发挥。

---

## 💡 核心理念

### 自主 ≠ 无结构

| 错误理解 | 正确理解 |
| :--- | :--- |
| 让 LLM 自己决定控制流 | 控制流在 Runtime，LLM 只做决策 |
| 外部画 DAG 工作流 | 内部有极小控制内核 |
| 无限自由发挥 | 在边界内自由而不失控 |

### 关键原则

1. **控制流在 Runtime，不在模型**
2. **副作用走 Job，写入走 Patch，必须可回滚**
3. **并发产出建议，串行写入资源（单写多读）**
4. **任何长任务都要有安全点、可中断、可恢复**
5. **成功可固化为 Skills；严格可固化为 Scheme**

---

## 🆚 与现有方案的差异

| 维度 | LangChain/LlamaIndex | Dify/Coze | gm-agent |
| :--- | :--- | :--- | :--- |
| **控制流** | 代码定义 | 可视化 DAG | 事件驱动状态机 |
| **持久化** | 无/需自行实现 | 平台托管 | Durable (内置) |
| **恢复能力** | 无 | 依赖平台 | Checkpoint + 重放 |
| **语言** | Python | 无代码 | **Go** |
| **部署** | 自建 | SaaS | **本地优先** |

## 🎨 应用场景

### 第一阶段：个人生产力
- 代码重构助手
- 文档写作助手
- 知识整理助手

### 第二阶段：团队协作
- 多人任务分派
- 代码审查自动化
- 文档生成流水线

### 第三阶段：企业级
- 与 CI/CD 集成
- 多 Agent 协作
- 审计与合规

---

## 🏆 愿景

> 真正的自主系统像一个交通系统：
> - 规则清晰（Policy、锁、预算）
> - 可解释（Trace）
> - 可恢复（Durable）
> - 可积累（Skills/Schemes）
>
> 在这个边界内，Agent 才能自由而不失控。
